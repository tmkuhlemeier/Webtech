Cache Control

As mentioned earlier, browsers and servers have caching policies. When, where, and for how long a web item is cached depends on so called cache-control headers. These are HTTP-headers which are being added by a request of the client or by a response of a server. These headers are also called cache directives. [1]

Table 1. Some examples of these cache request and response directives. [2]
Request directives Response directives
max-age= <seconds>:
Keeps track of the time during which an item is "fresh". 		must-revalidate:
When a response is stale, the cache is not allowed to use the copy before it has been revalidated by the server.
max-stale[=<seconds>]:
Indicates that the client accepts a stale response. Has an option to change the maximum time a response is stale. 		public:
The response can be cached anywhere.
no-store:
Prevents a response being cached by whichever cache. Useful for e.g. sensitive information like bank account numbers. 		private:
The response can only be cached by the client.
no-cache:
A cached response must always be validated at the origin server.
only-if-cached:
The client requests a response from only a cache, without accessing the origin server.

The foregoing shows that the cache policy of a browser or server also influences how fast a web item is loaded in a browser. For example, if an item has a no-store cache-control header, it must always be retrieved from the origin server. On the other hand if, for example, the client accepts a “stale” item, then it can simply be fetched from a cache if it exists there.
